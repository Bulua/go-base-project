# 11-后端开发规范

## 1. 技术选型

后端建议使用 Go + Gin + GORM + MySQL + Redis + JWT + Viper + Zap。第一阶段优先保证认证、权限、审计和基础配置可闭环，不引入过重的微服务拆分。

## 2. 推荐目录

## 3. 分层约定

业务代码按 `handler -> service -> repository -> model` 组织。

```text
handler：HTTP 入参解析、参数校验、响应封装
service：业务规则、事务编排、权限判断
repository：数据库访问和查询条件构建
model：GORM 模型和领域常量
```

模块仍然存在，只是作为每个类型目录下的二级目录。例如用户管理相关文件分布为：

```text
internal/handler/user
internal/service/user
internal/repository/user
internal/model/user
```

跨模块调用优先通过 service 接口，不直接跨模块操作 repository。通用能力沉到 `pkg` 或 `internal/infra`，避免模块互相穿透。

## 4. 统一响应

接口响应建议统一为：

```json
{
  "code": 0,
  "message": "ok",
  "data": {},
  "trace_id": "request-trace-id"
}
```

分页响应建议统一为：

```json
{
  "items": [],
  "total": 0,
  "page": 1,
  "page_size": 20
}
```

## 5. 中间件顺序

```text
Recovery -> RequestID -> Logger -> CORS -> BodyLimit -> JWT认证 -> API鉴权 -> 操作审计 -> Handler
```

登录、刷新 Token、健康检查等接口通过 `gbp_api_skip_rules` 跳过鉴权。操作审计需要对密码、Token、密钥、身份证号等字段脱敏。

## 6. 事务边界

以下操作必须使用事务：

1. 创建用户并分配角色
2. 修改用户角色
3. 删除菜单并清理按钮、路由参数、授权关系
4. 角色菜单、按钮、API、数据权限授权
5. 字典主表和字典项的级联删除

## 7. 权限约定

API 权限资源键统一为：

```text
METHOD:/api/v1/resource
```

例如：

```text
GET:/api/v1/users
PUT:/api/v1/roles/:id/menus
```

鉴权时先检查白名单，再校验 JWT，再查询角色权限策略。`deny` 策略优先于 `allow` 策略；没有显式授权时默认拒绝。

## 8. 安全底线

1. 密码只保存哈希值，推荐 bcrypt 或 argon2id。
2. JWT 黑名单只保存 Token 摘要，不保存原始 Token。
3. 登录失败、退出登录、重置密码和权限变更必须记录审计日志。
4. 管理员账号、超级角色和系统内置菜单需要保护，避免被普通接口删除。
5. 所有列表接口默认分页，并限制 `page_size` 最大值。
